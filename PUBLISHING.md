# Scheduled Publishing Pipeline

Write articles ahead of time, set a `publishAt` date in their frontmatter, and a GitHub Actions cron will release each one to your site + Dev.to + Hashnode + Medium + LinkedIn at exactly the right moment. You don't need to monitor anything once it's set up.

## How it works

```
content/articles/                    ←  your queue of pre-written articles
   ├─ satellites-to-autonomy.md          publishAt: 2026-05-20T09:00:00-07:00
   ├─ sysml-v2-ci-cd.md                  publishAt: 2026-06-03T09:00:00-07:00
   └─ mbse-at-startup-cadence.md         publishAt: 2026-06-17T09:00:00-07:00

content/published.json               ←  tracks what has shipped where
                                         (auto-committed by the workflow)

.github/workflows/scheduled-publish.yml
        │
        │  Cron: once a day at 9:05 AM Pacific
        ▼
scripts/run-scheduled-publish.js
        │  For each article where publishAt <= now AND not yet published:
        │
        ├──▶  generate app/writing/<slug>/page.tsx  →  Vercel auto-deploys
        ├──▶  publish-devto.js
        ├──▶  publish-hashnode.js
        ├──▶  publish-medium.js
        └──▶  publish-linkedin.js
        │
        ▼
   Updates content/published.json, commits + pushes back to main
```

## Setting `publishAt`

Use ISO 8601 with timezone:

```yaml
publishAt: 2026-06-03T09:00:00-07:00   # June 3, 9am Pacific
publishAt: 2026-07-15T14:30:00Z        # July 15, 2:30pm UTC
```

If `publishAt` is missing, the article stays in the queue but is never published. Useful for drafts.

## Scheduling a batch of articles

The fastest workflow:

1. Write 6–12 articles over a focused weekend
2. Set their `publishAt` values 2 weeks apart (or whatever cadence)
3. Push everything in one commit
4. Forget about it for 3+ months while the cron releases them on schedule

Example queue setup for a quarterly cadence of one post every two weeks:

```
content/articles/
├─ satellites-to-autonomy.md      publishAt: 2026-05-20T09:00:00-07:00
├─ sysml-v2-ci-cd.md              publishAt: 2026-06-03T09:00:00-07:00
├─ requirements-traceability.md   publishAt: 2026-06-17T09:00:00-07:00
├─ mbse-at-startup-cadence.md     publishAt: 2026-07-01T09:00:00-07:00
├─ executable-models.md           publishAt: 2026-07-15T09:00:00-07:00
└─ av-vs-aerospace-rigor.md       publishAt: 2026-07-29T09:00:00-07:00
```

## Article file format

```markdown
---
title: Your headline here
description: One-sentence summary for OG tags and platform descriptions.
slug: your-slug
publishAt: 2026-06-03T09:00:00-07:00
tags: [systems-engineering, mbse, sysml]
linkedinHook: |
  A more conversational hook for LinkedIn that goes above the link preview.

  Can be multiple lines. Will be used as the LinkedIn post body.
---

Your article body in plain markdown.

## Section headings use ## (h2)

Paragraphs are separated by blank lines.

The scheduler generates a basic Next.js page from this content automatically.
If you want richer formatting (figures, code blocks, callouts), create the
app/writing/<slug>/page.tsx by hand BEFORE the scheduled time. The scheduler
will skip page generation if the file already exists.
```

## What gets auto-generated vs. hand-written

**Auto-generated by the scheduler at publish time:**
- `app/writing/<slug>/page.tsx`: basic article page (paragraphs + h2 only)
- The Dev.to, Hashnode, Medium posts
- The LinkedIn share

**You write by hand:**
- `content/articles/<slug>.md`: the source of truth
- Optionally: a custom `app/writing/<slug>/page.tsx` if you want rich formatting

**You update manually after each release** (because it's nice to have):
- `app/page.tsx` and `app/writing/page.tsx`: the listing pages
- `app/sitemap.ts`: the sitemap

(These could also be automated later; for now they're manual because they affect site layout.)

## Initial setup (one time)

### 1. Push the repo to GitHub

```bash
git init
git add .
git commit -m "Initial commit"
git remote add origin git@github.com:<your-username>/stephanclaxton-site.git
git branch -M main
git push -u origin main
```

### 2. Deploy to Vercel

Visit https://vercel.com/new, import the repo, accept defaults. Your site is live in ~60 seconds.

### 3. Add platform secrets

In your GitHub repo: **Settings → Secrets and variables → Actions**

Add the ones you have, leave the rest empty (the scheduler skips platforms with missing credentials):

| Secret | Required for | How to get it |
|---|---|---|
| `DEVTO_API_KEY` | Dev.to | dev.to → Settings → Extensions → API Keys |
| `HASHNODE_TOKEN` | Hashnode | hashnode.com/settings/developer |
| `HASHNODE_PUBLICATION_ID` | Hashnode | From the GraphQL endpoint (see below) |
| `MEDIUM_TOKEN` | Medium | medium.com/me/settings/security → Integration tokens |
| `MEDIUM_USER_ID` | Medium | GET `https://api.medium.com/v1/me` with the token |
| ~~`LINKEDIN_TOKEN` / `LINKEDIN_AUTHOR_URN`~~ | LinkedIn (manual) | **Intentionally not automated** — see below |

The scheduler runs successfully even if only one platform is configured. Start with Dev.to (5-minute setup).

### LinkedIn is intentionally manual (with automatic reminders)

The `publish-linkedin.js` script exists and works, but it's not wired up to the cron by default. Reasons:

1. **LinkedIn requires a Developer App tied to a Company Page** for personal posting via API. Free to create but adds maintenance surface.
2. **OAuth tokens expire every 60 days**, requiring either manual re-auth every ~50 days or refresh-token rotation (not currently implemented).
3. **2x/month publishing cadence doesn't justify the API maintenance.** 30 seconds of manual copy-paste per article is cheaper than maintaining LinkedIn API tokens.

**How you get reminded:** When the cron publishes a new article to the site, it automatically creates a GitHub Issue labeled `linkedin-reminder`. The issue contains the article title, canonical URL, and the full `linkedinHook` text from the frontmatter, ready to copy-paste. GitHub sends you an email notification for the new issue. Open LinkedIn, paste the hook and URL, and close the issue.

If you ever want to flip on full automation, set `LINKEDIN_TOKEN` and `LINKEDIN_AUTHOR_URN` in GitHub Secrets. The script will start posting automatically and no reminder issues will be created. To get the credentials:

- App + OAuth: https://www.linkedin.com/developers/apps → Create app → request "Share on LinkedIn" + "Sign In with LinkedIn using OpenID Connect" products → OAuth scopes: `openid profile email w_member_social` → manual token exchange via cURL
- Author URN: `curl -H "Authorization: Bearer $TOKEN" https://api.linkedin.com/v2/userinfo` → format as `urn:li:person:<sub>`

### 4. Enable the workflow

GitHub disables scheduled workflows on new repos until they're committed to the default branch. After your first push, go to **Actions → Scheduled publish → Enable workflow**.

### 5. Done

The cron now runs every hour. When `publishAt` hits, your article ships to every configured platform, and `published.json` records what happened.

## Operating the queue

### Add a new article

```bash
# Write the file
vim content/articles/my-new-article.md   # include publishAt in frontmatter

# Commit and push
git add content/articles/my-new-article.md
git commit -m "Queue: my-new-article"
git push

# Nothing else needed. The cron picks it up at publishAt.
```

### Cancel a scheduled article

```bash
# Either delete the file:
rm content/articles/my-new-article.md
# Or comment out / remove the publishAt field:
# publishAt: 2026-06-03T09:00:00-07:00   <- removed
git commit -am "Cancel: my-new-article"
git push
```

### Reschedule

Change the `publishAt` value and push. As long as `content/published.json` doesn't yet have entries for that slug, it'll publish at the new time.

### Force-publish now (bypass schedule)

**Actions → Scheduled publish → Run workflow → Force-publish a specific slug now: `my-article-slug`**

This publishes the article immediately regardless of its `publishAt` date. Only the specified slug is processed; all other articles are ignored for that run.

### Re-publish an already-shipped article

Edit `content/published.json` to remove the entry for that slug + platform, commit, and the next cron run will re-publish that platform. Useful if something failed silently.

## Monitoring (the part you wanted to avoid)

The point of this design is you don't *need* to monitor it. But if you want to:

- **LinkedIn reminder issues:** When articles publish, the workflow creates a GitHub Issue with the LinkedIn hook text and URL. GitHub emails you about new issues, so this doubles as your "something published" notification.
- **Email on failure:** GitHub already emails you when an Actions workflow fails.
- **Per-run summary:** Each cron run writes a summary visible at Actions → Scheduled publish → \<run\> → Summary
- **Audit trail:** `content/published.json` is the canonical record of what shipped when, committed to git

If a platform's API fails for a given article, the workflow fails loudly (email), the state file stays unmodified for that platform, and the *next* daily run will retry it. No manual intervention needed for transient failures.

## Why this beats Buildkite, Vercel cron, or external schedulers

- **Buildkite** wants $30+/mo and needs an agent. Overkill for a cron.
- **Vercel cron jobs** are good for hitting your own API routes but don't have great "commit back to the repo" ergonomics. They'd also need a separate secrets store.
- **GitHub Actions cron** is free, runs in the same place your code lives, can commit back to the repo with the built-in `GITHUB_TOKEN`, and has decent reliability for hourly cadence.

Tradeoff: GitHub Actions cron can be delayed by 5–15 minutes during periods of high platform load. For "publish at exactly 9:00:00 AM" you'd need a real scheduler. For "publish sometime in the 9 AM hour" (which is what reputation-building publishing actually needs) Actions is plenty.

## Why daily, not hourly

The cron runs once a day at 9:05 AM Pacific. Earlier versions of this pipeline ran hourly, which was wasteful. Most runs were no-ops because no article was due. Daily covers the actual use case (articles are scheduled to specific dates, not specific hours) while using ~24x less compute.

If you want articles to publish at a different hour, change the cron expression in `.github/workflows/scheduled-publish.yml`. The format is `minute hour * * *` in UTC.

If you ever need an article to publish at a precise hour that's not the daily slot, set its `publishAt` to the future and use the `workflow_dispatch` button (Actions → Scheduled publish → Run workflow) at the exact moment you want it to ship.

## What this pipeline deliberately does NOT do

- **No engagement automation** (auto-liking, auto-following). Bannable, ineffective, and creates the exact footprint you don't want.
- **No content generation.** Every article is something you wrote, which is the whole point.
- **No spinning/rewording** for different platforms. Google penalizes near-duplicate content; canonical URLs are the right pattern.
- **No Twitter/X.** Requires paid API tier ($200/mo minimum to post). Easy to add as `publish-x.js` later if you want it.
- **No Instagram / TikTok.** Meta's Business API approval process is multi-week and personal accounts can't post via API.
