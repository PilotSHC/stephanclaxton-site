# stephanclaxton.com

Personal site for Stephan Claxton. Systems engineering for AI and autonomy, with prior work in space and defense.

Built with Next.js 14 (App Router). Articles are authored as Markdown in `content/articles/`, with a scheduler that releases them on a daily cron and cross-publishes to Medium, Dev.to, Hashnode, and LinkedIn.

## What's in the box

- Editorial-minimal design (Fraunces + JetBrains Mono, paper-warm palette)
- Schema.org `Person` markup in the root layout for SEO disambiguation
- Per-article `Article` schema for richer search results
- Auto-generated sitemap and robots.txt
- Open Graph + Twitter card metadata
- Markdown-authored articles with frontmatter (`publishAt`, `title`, `description`, …)
- Cron-driven scheduled publish + state tracking in `content/published.json`

## Run it offline

Prerequisites: Node 18+ (you currently have v25.x), npm 9+.

```bash
# 1. Install site dependencies
npm install

# 2. (Optional) install publisher script dependencies. Only needed if you want
#    to test the cross-posting scripts locally.
cd scripts && npm install && cd ..

# 3. Start the Next.js dev server
npm run dev
# Serves http://localhost:3000 (falls through to :3001 if 3000 is busy)
```

The dev server has hot-reload. Edit any file under `app/`, `content/`, or
`scripts/` and the page refreshes automatically. No database, no external
services required to develop or preview locally.

### Production-style local run

```bash
npm run build   # compiles to .next/
npm start       # serves the production build on :3000
```

### Useful commands

```bash
npm run lint    # next lint
rm -rf .next    # nuke the build cache if you ever see stale content
```

## Adding a new article

1. Create a Markdown file at `content/articles/<slug>.md` with frontmatter:

   ```markdown
   ---
   title: "Title of the piece"
   description: "One-sentence dek for SEO and link previews."
   publishAt: "2026-06-15T16:00:00Z"   # when the scheduler should release it
   tags: [mbse, sysml]
   ---

   Body in Markdown.
   ```

2. Commit and push. The `scheduled-publish.yml` workflow runs daily at
   16:05 UTC. When `publishAt` has passed, it will:
   - Mark the article published in `content/published.json`
   - Generate the writing page under `app/writing/<slug>/`
   - Cross-post to Medium, Dev.to, Hashnode, and LinkedIn
   - Commit the state change back to `main`

3. To force-publish a slug immediately (bypassing `publishAt`), trigger the
   workflow manually from the GitHub Actions tab with the `force_slug` input.

## Cross-publish secrets (GitHub Actions)

The scheduler reads these from repo secrets:

| Secret | Used by |
| --- | --- |
| `MEDIUM_TOKEN`, `MEDIUM_USER_ID` | `publish-medium.js` |
| `DEVTO_API_KEY` | `publish-devto.js` |
| `HASHNODE_TOKEN`, `HASHNODE_PUBLICATION_ID` | `publish-hashnode.js` |
| `LINKEDIN_TOKEN`, `LINKEDIN_AUTHOR_URN` | `publish-linkedin.js` |

For local testing, drop them into a `scripts/.env` (gitignored) and source it
before running individual `node scripts/publish-*.js` commands. See
`PUBLISHING.md` for the full setup guide.

## Deploy to Vercel

1. Push this repo to GitHub
2. Go to https://vercel.com/new
3. Import the repo, accept defaults, click Deploy
4. In Vercel project settings, add the custom domain `stephanclaxton.com`
5. Update your domain registrar's records per Vercel's instructions

No build configuration needed.

## Files of note

```
app/
  layout.tsx                ← site chrome + Person JSON-LD
  page.tsx                  ← homepage
  about/page.tsx            ← about page
  writing/page.tsx          ← writing index
  writing/<slug>/page.tsx   ← generated by the scheduler at publish time
  globals.css               ← all styles
  sitemap.ts
  robots.ts
content/
  articles/<slug>.md        ← author markdown sources
  published.json            ← scheduler state (which slugs went live, when)
scripts/
  run-scheduled-publish.js  ← the scheduler entrypoint
  publish-medium.js
  publish-devto.js
  publish-hashnode.js
  publish-linkedin.js
  lib/load-article.js       ← markdown + frontmatter parser
  lib/published-state.js    ← published.json reader/writer
.github/workflows/
  scheduled-publish.yml     ← daily cron (16:05 UTC)
```

## Why this stack

- Next.js gives server-rendered HTML out of the box, so Google indexes it faster and more reliably than client-rendered SPAs
- The App Router's metadata API makes per-page SEO trivial
- Vercel's free tier covers a personal site indefinitely
- Markdown sources + a scheduler means you write once, commit, and the same canonical content propagates to every external platform with a `canonical_url` pointed back here
